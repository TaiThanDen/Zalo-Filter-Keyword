You are the implementation agent for this project.

You have been given a VPS handoff bundle and must use it as the source of truth for environment access, safety rules, execution order, and acceptance criteria.

YOUR JOB
Read the full handoff bundle first, then SSH into the VPS, inventory the environment, protect the existing Zalo browser session, and set up the project end-to-end as far as safely possible.

NON-NEGOTIABLE RULES
1. Do not log out Zalo.
2. Do not clear cookies, local storage, or browser profile data.
3. Do not delete or overwrite the current Chromium/Zalo profile.
4. Do not reboot the VPS unless explicitly allowed by filled values.
5. Do not make risky browser changes before backing up the active profile.
6. Keep route handlers thin and business logic outside React pages.
7. If the real watcher is unsafe or unclear, stop at watcher simulator and document the blocker clearly.
8. Always prefer reversible and idempotent changes.

READ IN THIS ORDER
1. README.md
2. docs/00-handoff-summary.md
3. docs/01-vps-access-and-safety.md
4. docs/02-current-state.md
5. docs/03-target-state.md
6. docs/04-agent-execution-plan.md
7. docs/05-acceptance-checklist.md
8. docs/06-values-to-fill.md

THEN EXECUTE IN THIS ORDER

PHASE A — ACCESS + INVENTORY
- SSH into the VPS using the provided SSH values.
- Immediately collect:
  - whoami
  - hostname
  - pwd
  - uname -a
  - OS release
  - disk space
  - memory
  - browser processes
  - Node/npm/pnpm/git versions
  - display/session context
  - browser executable and version
  - current browser user-data-dir/profile path if discoverable
- Do not make destructive changes in this phase.

PHASE B — BACKUP + SAFETY
- Back up the current Chromium/Zalo profile before changing anything browser-related.
- Back up relevant repo/config/service files before editing.
- Record rollback notes.

PHASE C — REPO + APP SETUP
- Locate existing repo or clone it if needed.
- Install dependencies.
- Set up env values.
- Run migrations/seeds if applicable.
- Get the Next.js app running.
- Get worker runtime running.

PHASE D — CORE FEATURES
Implement or finish:
- auth
- groups CRUD
- rules CRUD
- channels CRUD
- ingest API
- normalize + dedupe + rule engine
- logs
- Telegram notification flow
- watcher heartbeat/config endpoints

PHASE E — WATCHER SIMULATOR
- Ensure simulator runs end-to-end.
- Prove the full flow:
  login/admin config -> simulator ingest -> match -> delivery -> logs -> watcher status

PHASE F — REAL WATCHER (ONLY IF SAFE)
Only proceed if:
- active profile was backed up,
- attach/launch strategy is clear,
- rollback path is clear,
- real watcher setup is allowed by filled values.

If not safe, stop at simulator and document:
- what is missing,
- what is risky,
- what exact next step a human should approve.

MANDATORY OUTPUT FORMAT AFTER EACH PHASE
1. What changed
2. Files added/updated
3. VPS changes made
4. Safety/backup status
5. How to run/test
6. Known limitations
7. Next step

FINAL SUCCESS BAR
You may only claim success when:
- SSH access was verified,
- current browser/profile state was documented,
- backups were taken before risky changes,
- app and worker run,
- simulator works end-to-end,
- and if you touched the real watcher, you can show the Zalo session still works.

Start by:
1. confirming you have read the full handoff bundle,
2. listing all unresolved values from docs/06-values-to-fill.md,
3. stating your SSH/access plan,
4. then beginning PHASE A only.
